# 自然指南针 (Nature Compass) 内容生成系统：修复与优化实施方案

## 1. 执行摘要

对《内容评审报告》和《套件优化计划》的分析揭示了两类不同但相互关联的故障：

1. **灾难性系统故障 (Catastrophic System Failure)：** 生成管道产生 `undefined` 的内容，并在特定请求上提示 `fetch failed`（参见错误日志）。这是一个 P0 级（最高优先级）、阻断流程的 Bug，直接导致产品无法使用。
2. **系统性质量缺陷 (Systemic Quality Failure)：** 即便内容成功生成，也存在严重的教学和安全缺陷，主要体现在：缺乏室内备选方案、节奏把控不佳、教师指导不足。这些是源于底层 AI 提示词和输出验证不足的 P0 级产品级缺陷。

本文档概述了一个具体的、多阶段的实施方案来解决这些问题。我们的策略是：首先通过修复内容生成 Bug 来**稳定 (Stabilize)** 系统，然后通过实施 P0 级提示词和结构更改来**修复 (Remediate)** 核心质量问题，最后通过 P1/P2 级改进来**增强 (Enhance)** 产品。该计划强调安全性、可行性以及建立可扩展的、用于全局管理提示词和 UI 更改的架构。

## 2. 指导原则

* **安全与可用性优先：** 所有更改必须优先考虑学生的安全和教育者的可用性。缺乏室内替代方案直接构成了安全和操作风险。
* **系统性修复优于临时补丁：** 我们将通过建立强大的提示词管理系统和 AI 生成内容的数据契约来解决根本原因，而不是对单个课程进行修补。
* **数据驱动的验证：** 所有更改都将根据报告中列出的自动检查和专家评审评分进行衡量。成功的定义是消除 `undefined` 错误并显著提高质量指标。
* **分阶段和灰度发布：** 任何更改都不会在没有经过自动化验证、AI 辅助审查和有限灰度发布（Canary Release）的情况下直接 100% 部署给用户。

## 3. 多阶段实施方案

### 阶段 0：分诊与系统稳定 (立即执行：未来 72 小时)

此阶段完全专注于灾难性系统故障。目标是恢复基本功能并查明产生 `undefined` 内容的根本原因。

| 任务 | 负责人 | 描述 | 可行性与安全性 |
| :--- | :--- | :--- | :--- |
| **0.1: 根本原因分析 (RCA)** | 首席工程师, AI 架构师 | 成立事件响应小组调查 `fetch failed` 错误和 `undefined` 内容。假设包括：API 超时、模型拒绝回答（触发安全过滤）、响应格式错误或前端渲染 Bug。 | **可行性高。** 属于标准调试流程。**安全性：** 主要风险是系统中断时间延长，因此立即调查是最安全的路径。 |
| **0.2: 实施临时缓解措施** | 前端/后端团队 | 如果问题是间歇性的，在 UI 中实现健壮的错误处理，显示用户友好的消息（“内容生成中，请重试”）而不是带有 `undefined` 的崩溃页面。如果问题持续存在，暂时禁用相关类型套件的生成。 | **可行性高。** 防止用户直接暴露于崩溃状态。这是保护品牌声誉和用户信任的关键安全网。 |
| **0.3: 部署紧急修复 (Hotfix)** | 工程团队 | 基于 RCA 结果，部署 Hotfix 以解决导致内容生成失败的直接原因。 | **可行性取决于具体原因。** 视根本原因复杂程度而定。**安全性：** 修复必须在 Staging 环境中彻底测试，以防引入回归。 |

**成功标准：** 生产环境中零 `undefined` 内容渲染。所有套件生成请求要么成功，要么优雅地失败并提供面向用户的反馈。

### 阶段 1：基础修复 (P0 级修复 - 冲刺 1-2)

此阶段解决两份报告中指出的最核心的质量和安全问题，需要提示词工程和后端架构的协同努力。

| 任务 | 负责人 | 描述 | 可行性与安全性 |
| :--- | :--- | :--- | :--- |
| **1.1: 建立集中的提示词注册表** | AI 架构师, 后端团队 | 创建一个版本控制系统来管理所有 AI 提示词（详见第 4 节）。这是系统性更改的先决条件。 | **中等可行性。** 需要前期架构工作，但对可扩展性和可维护性至关重要。直接风险低。 |
| **1.2: 整合 P0 级提示词变更** | 提示词工程师, AI 架构师 | 在新的注册表中，使用 **P0 建议**（室内备选方案、时长与节奏、语言/词汇支架）更新核心课程生成提示词。这些指令必须作为模型的强制要求。 | **高可行性。** 提示词文本已明确。**安全性：** 需要严格测试以确保模型能可靠地遵循这些新的复杂指令。 |
| **1.3: 强制执行严格的输出 Schema 校验** | 后端团队, AI 架构师 | 修改生成服务，使用严格的数据契约（如 JSON Schema）验证 AI 的输出。Schema **必须** 要求存在 `indoor_alternative_plan`、`teaching_tips`、`learning_objectives` 和 `assessment_rubric` 等字段。任何验证失败的生成都必须被视为系统错误并进行重试或抛出异常，绝不能发送给用户。 | **中等可行性。** 这是一项关键的架构改变。它将“内容质量”问题转化为可测试的“系统完整性”问题，是消除内容缺失段落最有效的方法。 |
| **1.4: 重新生成并验证所有套件** | QA, 课程团队 | 使用新的提示词和 Schema 验证，触发所有课程套件的批量重新生成。所有重新生成的内容必须通过新的、更严格的自动化验证套件。 | **高可行性。** 属于操作性任务。**安全性：** 重新生成的内容必须在 Staging 环境中进行审查，防止语气或质量出现不可预见的负面变化，然后再推向生产环境。 |

**成功标准：**

* 100% 的生成户外课程包含非空的室内备选方案 (`indoor_alternative_plan`)。
* 教师提示 (`teaching_tips`) 部分包含了可操作的详尽指导。
* 自动化字数检查在关键页面上不再失败。
* ESL 课程包含明确的语言支架（Language Scaffolding）元素。

## 4. 全局应用策略 (提示词 & UI 变更)

系统化地应用这些变更对于避免技术债务和各组件之间的不一致至关重要。

### 提示词管理：“Prompt as Code” 注册表

我们将彻底停止将提示词作为松散的字符串硬编码在代码库中。所有提示词将在专用的、受版本控制的库中进行系统管理。

| 组件 | 描述 | 优势 |
| :--- | :--- | :--- |
| **提示词模板化** | 提示词将使用模板语言 (如 Jinja, Handlebars) 存储在单独的文件中，允许动态传入变量如 `age_group`、`cefr_level` 等。 | **复用性与一致性：** 单个模板可服务于上百节课程，确保所有生成内容无缝遵循相同的核心指令。 |
| **版本控制 (Git)** | 提示词库将由 Git 全面接管。对提示词的更改需要像生产代码一样提交 Pull Request 并经历代码审查阶段。 | **可审计性与回滚：** 我们可以准确追踪每次提示词的更改，了解其爆炸半径（Blast Radius），并在质量下降时瞬间回滚到先前的稳定版本。 |
| **CI/CD 自动化** | CI/CD 管道将自动进行冒烟测试并部署新的提示词版本到 Staging 环境进行自动化验证，然后再推向生产环境。 | **自动化与安全：** 防止人为失误，并确保所有提示词在触达真实用户前通过严格验证。 |

### UI 代码变更：组件驱动架构

所有 UI 更改都将采用集中式的、基于组件的前端框架进行处理。

* **坚守数据契约：** 前端设计将被限制为仅消费阶段 1 中定义的严格 JSON Schema。如果向 AI 输出添加了新字段（例如 `indoor_alternative_plan`），则必须显式创建一个对应的 `<IndoorAlternativePlan>` 组件。
* **单一事实来源 (Single Point of Change)：** 要添加或修改某块内容的展示方式，开发人员只需更改该特定组件。此更改随后将自动反映在应用该组件的所有界面中。例如，改善 `<TeachingTips>` 组件的渲染逻辑，整个平台中所有的“教师提示”都将统一升级。
* **防御性条件渲染：** UI 将按最坏情况进行防御性编程。如果数据中的某个可选字段碰巧为空，组件将平滑回退（Fallback）或折叠，绝不会阻断整个页面的渲染链（例如 ` {data.pre_lesson_activity && <PreLessonActivity content={data.pre_lesson_activity} />} `）。这与后端的严格 Schema 验证相辅相成，构成双重保障。

## 5. 可行性与安全性评估

该计划目标宏大但路线切实可行。首要风险是大型语言模型 (LLM) 固有的非确定性和引入质量回归的可能性。我们的安全协议专门为应对这些风险而设立。

**适用于核心更改的安全与验证全流程卡点：**

1. **离线评估 (Offline Evaluation)：** 首先使用拟议的提示词变更生成 100-200 套课程作为“黄金基准测试集”，将它们的各项指标与当前的生产版本基线进行比较。
2. **自动化防御关卡：** 新生成的测试套件必须 100% 绿灯通过所有自动化检查（字数限制、强制包含节点、Schema 反序列化检查等）。
3. **大模型交叉审核 (Post-Phase 2)：** 由扮演“审查官”角色的次级 LLM 对新数据集进行打分，提前侦测体验降级。
4. **灰度发布 (Canary Rollout)：** 将更改部署到仅仅 1% 的真实请求流量。实时且密切地监视错误率、延迟和异常请求日志。
5. **人工抽样：** 课程专家团队将从灰度流量的输出中随机抽取样本进行体感评估。
6. **全量广播 (Full Deployment)：** 仅当上述所有的防线都被证实安全且指标优良后，更改才会推向所有用户节点的 100% 流量。

通过坚定地执行上述高度结构化、分阶段和防范未然的蓝图，我们可以有效根除 Nature Compass 平台中的系统性故障和致命的盲区，将其从一个部分残缺的实验性产品，锻造为一个极度稳定、极致安全、且富有非凡教育深度的教学引擎。

# 开发日志: Review Queue UX & Data Integrity Fixes

**日期**: 2026-03-16

## 🎯 重点完成任务 (Key Achievements)

### 1. Review Panel 重构 (页边批注模式)

- **痛点**: 原先的 Review Panel 嵌在页面最底部（或者用悬浮遮挡模式），导致教师需要一直滚动到底部才能看到，并且操作时极大地阻滞了直接阅读和修改 Lesson Plan 的视线。
- **解决方案**: 我们将其重构为 **页边批注 (Margin Annotation)** 布局。将 Toggle 开关栏高亮置于 Lesson Plan 顶部以便第一时间发现。展开后，面板利用 `BodyContainer` 右侧原本留白的屏幕边距进行绝对定位 (`fixed` 定位，根据内容区右边界计算 `left` 坐标)，从而实现了审查建议与原有教案的**左右并排无缝显示**。在空间不足的窄屏设备上，则自动优雅降级为传统遮罩浮层。
- **交互增强**: 针对每条审查结果，新增了 "Apply" (调用 AI 触发单阶段无缝修复) 和 "定位" (高亮闪烁并滚动到对应教学阶段表格) 的快捷行动按钮。

### 2. 跨组件导航闭环 (Review Queue -> Lesson Kit)

- **痛点**: 在左侧 Sidebar 的 "Review Queue" 中点击处于待审查状态的课程时，默认操作是跳转到泛化的全局 Records 记录列表页，无法形成审阅闭环。
- **解决方案**: 在通用的共享层组件 `AppLayout` 中增设了 `onFinderItemClick` 事件回调机制打断了传统的 A 标签锚点跳转。然后在应用级入口 `App.tsx` 中桥接这个回调，通过传入的 Record ID 从内存里（`savedLessons`）精确匹配到对应的课程详情，进而直接注入状态对象，并拉起 `CreatePage`（即可编辑的详情视图），实现了“所见即所得”的无缝切换。

### 3. 数据库时间戳一致性修复

- **痛点**: 所有的卡片展示的时间出奇一致，失去了时间线追溯的能力。
- **解决方案**: 深层剖析后发现，尽管底层 Supabase 的信封记录了准确的改动时间，但在 `recordRepositoryAdapter.ts` 通过 `fromLessonEnvelope` 反向序列化为内存可用模型时，没有把外层信封对象的审计关键字段 (`createdAt/updatedAt`) 解包并映射到 Payload 的核心字段 (`timestamp/lastModified`)。导致整个应用层仍旧复用一开始默认的时间。修复该映射后，新记录的时间线重回正轨。
- **遗留数据清理**: 在程序启动周期内，通过在 `App.tsx` 动态注入单轮脚本，全表洗刷并移除了带有 "Legacy" 后缀字样的陈旧幽灵课程。

## 💣 经常出错 / 踩坑点 (Common Pitfalls)

1. **巨型状态树下的深层局部修改**: `GeneratedContent` 这个大对象层级深且复杂（单元 > 阶段 > 操作步骤）。AI 提供局部的针对性修复后，如果状态更新使用简单的 `replace` 逻辑，极容易覆盖冲刷掉该阶段的其它未改动兄弟属性。必须确保针对 `stageIndex` 极其精细地进行合并克隆。
2. **DOM 同步计算与脱离文档流**: 把审查侧边栏挂载在外侧时，其相对参照坐标 (`left`) 高度依赖 React 渲染树里的其它实体节点（比如 `BodyContainer` 的当前渲染宽度）。遇到路由跳转或者屏幕缩放的情形下，这种测量可能会闪烁失真甚至定位错位，我们不得不在组件 Resize 或渲染周期内动态计算。
3. **ORM 映射时数据视角的断层 (Envelope vs Payload)**: Repository 架构为了通用性，把对象分为了承载业务数据的 Payload 和承载鉴权/时间的 Envelope。日常开发极易过度偏向于操作 Payload，而在存读操作里产生属性不互通的错觉，导致类似时间戳不同步的奇怪现象。

## 📝 未完成 / 遗漏的任务 (Pending / Omitted Tasks)

1. **脏数据的逆向溯源问题**: 时间戳由于之前已经固化在了各种云平台和本地库上，对于那些历史已经丢失准确时间的课程卡片，如果不被强制从头执行一遍“改写并保存”(Trigger DB `updated_at` Refresh)，其旧错版时间依然会存在。
2. **Lint 和辅助功能的噪音**: 控制台终端依然在报多条无关紧要的 ESLint 和可访问性检查（比如某些原有的 `select` 标签或删除 `button` 没有加 Title、ARIA attr，以及少量的 inline style 排版），这部分暂缓了清理。

## 🚀 接下来可继续进行的开发方向 (Next Steps)

1. **更细微的审计反馈 (Human-in-the-loop Diff View)**:
   虽然现在提供了 “Apply” 自动采纳修改功能，但是对于核心教育环节，直接替换可能不够透明。接下去可以在点击 “Apply” 后呈现一个小型的 Diff 预览面板，让教研老师看到“增删查改”的具体对比，确认无误后再融合。
2. **大纲 (Curriculum) 层级的审阅面板延伸**:
   当前的自检引擎都挂载在单节具体“Lesson Plan”里。对于整个大纲目前还没有宏观层级的交叉审查（例如：不同课时之间语法焦点跳跃是否太大、是否缺乏复习环路）。可以把该审查架构复刻并抽象一套高维引擎给大纲使用。
3. **技术债清理大扫除**:
   找整个完整的回合，全量修复一下整个项目所有的 a11y 警告与 TypeScript 报错。
